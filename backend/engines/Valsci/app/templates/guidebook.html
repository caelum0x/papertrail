What Is Valsci?

Automated scientific claim verification powered by LLMs and academic literature.

Valsci is a tool that evaluates scientific claims against published research. You give it a claim like "Creatine supplementation improves short-term memory" and it:

  • Generates search queries from the claim using an LLM.
  • Retrieves academic papers from Semantic Scholar matching those queries.
  • Analyzes each paper's relevance to the claim, extracting key excerpts and evidence.
  • Scores paper reliability using bibliometric data (author H-index, citation counts, journal quality).
  • Produces a final report with a verdict rating from "Contradicted" to "Highly Supported," backed by cited evidence.

Every step is grounded in real published papers—not LLM imagination. The system records a detailed trace of every decision so you can audit exactly how it reached its conclusion.

Two Ways to Evaluate

Valsci offers two evaluation modes, each suited to different needs:

  • Quick Runs (Home page) — Submit one or more claims and evaluate them with a single provider and model. Fast and straightforward. Best when you trust a particular model or just want a quick answer.
  • Arenas (Arena Builder) — Submit the same claims to multiple models simultaneously and compare their results side-by-side. Best when you want to understand how different LLMs assess the same evidence, or when you need high confidence in the verdict.

How Costs Work

Every claim evaluation consumes LLM tokens (the units that language models charge for). The cost depends on:

  • The model — Different models have different per-token prices. These are configured per model on the Providers page (Input Cost and Output Cost per million tokens).
  • The number of queries and papers — More search queries and more results per query mean more papers to analyze, which means more tokens.
  • Bibliometric scoring — When enabled, adds an extra scoring stage per paper.

Every submission starts with a preflight cost estimate that you review and approve before anything queues. The actual cost may differ slightly because token counts are estimates until the LLM responds. All costs are tracked per run and visible in reports, batch results, and arena workspaces.

Tip: If you're exploring locally, start with the curated mini corpus on the Data page. It keeps the production Semantic Scholar lookup path intact while using a small focused dataset.

Getting Started

Your first claim evaluation in five steps.

Step 1: Configure a Provider

Go to the Providers page. You need at least one enabled provider with a valid API key and at least one enabled model. If you're using OpenAI, enter your API key, ensure a model like gpt-4o is listed and enabled, and save. See the Providers section for full details.

Step 2: Submit a Claim

Go to the Home page. Select your provider and model from the dropdowns. Type a scientific claim (e.g., "Vitamin D supplementation reduces the severity of COVID-19") into the Claims textarea. Click Stage Claims.

Step 3: Review and Queue

Click Review & Queue Run. Valsci estimates the projected token usage and dollar cost, then opens a confirmation modal — approve the spend there and the run queues. You land on the live Progress page automatically.

Step 4: Monitor Progress

You'll be redirected to the Progress page, which shows a real-time progress bar as your claim moves through the pipeline stages. When complete, you'll be sent to Batch Results.

Step 5: Read the Report

Click View Report on your claim in Batch Results. You'll see the overall verdict, supporting evidence with paper excerpts, non-relevant papers, and full cost/timing breakdowns. From here you can download the report, view the execution trace, or re-run the claim with different settings.

Tip: Once you're comfortable with single-model runs, try creating an Arena to compare how different models evaluate the same claims. It's the best way to build confidence in your results.

Key Concepts

Core terminology used throughout Valsci.

Claim
A scientific assertion you want to verify, such as "Creatine supplementation improves short-term memory." Valsci evaluates claims by searching for published research and analyzing the evidence.
Run
A single evaluation of one claim by one provider and model combination. A claim can have many runs, each potentially using different models or settings, letting you compare how different LLMs assess the same claim.
Batch
A group of claims submitted together in one operation. Every submission from the Home page creates a batch, even if it contains only one claim.
Arena
A structured comparison where the same set of claims is evaluated by multiple provider/model candidates. Arenas progress through stages, letting you review intermediate results and decide which candidates advance.
Candidate
A specific provider + model combination participating in an arena. Each selected model becomes a named candidate with a consistent identity across all claims in the arena.
Provider
A connection to an LLM API service, such as OpenAI, Azure OpenAI, Ollama, or a local llama.cpp server. Each provider stores its credentials, base URL, and a list of available models.
Stage
One step in the claim evaluation pipeline. The four stages are:
  • Query Generation — The LLM reads your claim and generates multiple targeted search queries designed to find relevant academic papers. For example, the claim "Creatine improves memory" might produce queries like "creatine supplementation cognitive function randomized trial" and "creatine monohydrate working memory neuroimaging."
  • Paper Analysis — Each query is sent to Semantic Scholar. Retrieved papers are then individually analyzed by the LLM, which reads the title, abstract, and metadata to determine relevance to the claim and extract supporting or contradicting excerpts.
  • Venue Scoring — When bibliometrics are enabled, each paper receives a reliability score based on the authors' H-indices, the paper's citation count, and the journal's impact. Higher-quality sources carry more weight in the final verdict.
  • Final Report — The LLM synthesizes all analyzed papers, their excerpts, and their reliability scores into a narrative report with an overall claim rating.
Semantic Scholar
The academic paper database that Valsci searches to find evidence for claims. It returns published papers, their authors, citation counts, and other metadata.
Bibliometrics
Quantitative measures of paper and author quality, including author H-index, citation count, and journal impact. When enabled, these scores weight evidence so that higher-quality sources carry more influence in the final verdict.
Trace
A detailed, step-by-step log of every LLM call, API request, and processing decision made during a run. Useful for debugging, auditing prompt behavior, and understanding token usage.
Claim Rating
The final verdict for a claim, expressed on an ordinal scale:
  • 0 — No Evidence: No relevant papers were found. The claim can't be assessed.
  • 1 — Contradicted: The available evidence directly contradicts the claim.
  • 2 — Likely False: The weight of evidence leans against the claim, though some ambiguity exists.
  • 3 — Mixed Evidence: Papers both support and contradict the claim; no clear consensus.
  • 4 — Likely True: The weight of evidence supports the claim, with minor caveats.
  • 5 — Highly Supported: Strong, consistent evidence from multiple high-quality sources supports the claim.
Prompt Provenance
Valsci tracks exactly which prompt templates were used for each stage of every run, including file names and SHA-256 hashes. This ensures reproducibility: you can verify that two runs used identical prompts, or identify when a prompt change affected results.
Issues
Warnings or errors that occurred during processing, such as LLM timeouts, parsing failures, or context window overflows. Issues don't necessarily mean the evaluation failed—the system has retry and recovery logic—but they flag areas worth reviewing in the trace.

Navigation & Layout

How to get around the application.

Top Navigation Bar

The sticky bar at the top of every page contains:

  • VS / Valsci (brand lockup, far left) — Click to return to the Home page from anywhere.
  • Home — The claim submission page (quick runs).
  • Claims — The Claims Browser, where you search and browse all evaluated claims.
  • Arenas — The Arenas Library, listing all multi-model comparisons.
  • Providers — Provider configuration: API keys, models, and connection settings.
  • Migration — Legacy batch import tool (only relevant if upgrading from an older Valsci version).
  • Guidebook — This page.

Status Indicators

  • Logout button (top right) — Appears when password protection is enabled. Ends your session and redirects to the login page.

Page Structure

Most pages follow a consistent layout:

  • Breadcrumbs — Appear on sub-pages (like Claim Detail) showing the navigation path. Click any breadcrumb to go back to that level.
  • Page header — Title, optional subtitle, and optional action buttons (e.g., "Refresh" or "New Arena").
  • Panels — Content is organized in bordered panel cards. Some pages use a two-column dashboard grid layout (e.g., Home, Providers, Claim Detail).
  • Status cards — Colored alert banners (blue for info, yellow for warnings, red for errors, green for success) appear contextually to communicate system state.
  • Modals — Cost confirmation dialogs and import modals overlay the page. They always require explicit approval (a checkbox) before proceeding with cost-incurring operations.

Home Page

Submit claims for evaluation using a single provider and model.

Migration Alert

If legacy batch folders exist from an older version of Valsci, a yellow warning banner appears at the top of the page. It shows how many legacy batches need attention and offers two actions:

  • Review Migration — Opens the Migration page so you can inspect each legacy batch before importing.
  • Migrate & Archive — Immediately imports all pending legacy batches into the canonical claim store and moves their old folders into the migration archive.

Run Claims Panel

The main panel on the left side of the page is where you configure and submit claims.

Provider & Model

The Provider dropdown lists all enabled providers configured on the Providers page. Selecting a provider populates the Model dropdown with that provider's enabled models. These determine which LLM service and model will evaluate your claims.

Existing Claim Handling

The Existing Claim Handling dropdown controls what happens when you submit a claim that has already been evaluated:

  • Create New Run — Always creates a fresh evaluation, even if the same claim text already exists. Useful for comparing results across different models or settings.
  • Open Existing Claim — If the claim already exists, navigates directly to the existing claim detail page instead of running it again.

Batch Tags

An optional comma-separated list of tags to label this batch. Tags help you organize and identify batches later in the Claims Browser.

Claims Input

Enter claims in the Claims textarea, one claim per line. Alternatively, click Upload Claims to load claims from a .txt file (one claim per line).

Stage Claims / Clear

  • Stage Claims — Parses the textarea or uploaded file and moves the claims into the staging area below. Claims are not submitted yet; staging lets you review and edit before committing.
  • Clear — Removes all claims from both the textarea and the staging area.

Staged Claims

A list of claims ready to be submitted. You can review them here before proceeding. Staging is a safety step: it lets you verify claim text, remove duplicates, or catch paste errors before incurring any cost.

Advanced Run Settings

Click to expand the advanced settings panel:

  • Queries — The number of search queries the LLM will generate from each claim (default: 5). More queries cast a wider net but increase cost.
  • Results / Query — How many papers to retrieve per search query (default: 5). Higher values find more papers but increase processing time.
  • Use bibliometric scoring — When checked, papers are weighted by author H-index, citation count, and venue impact. Uncheck to treat all papers equally.

Cost Estimation & Submission

  • Review & Queue Run — A single two-step button. It first runs a preflight estimate of token usage and dollar cost (shown in the preflight panel below), then opens a confirmation modal. You must check the "I approve this estimated spend" checkbox before confirming. Once confirmed, claims are submitted for processing and you are redirected to the Progress page. If you change any settings after the estimate, the next click re-estimates first.

Data

Inspect, download, index, and verify local Semantic Scholar data.

Navigate here using the Data link in the top navigation bar. The Data page is where you prepare Valsci for real Semantic Scholar retrieval. It shows which releases are present locally, which datasets are indexed, whether the API key is configured, and what downloader or indexer job is currently running.

When to Use the Data Page

  • First real-data setup — Build the curated Mendelian mini corpus or download full Semantic Scholar datasets before running claims.
  • Pipeline rehearsal — Confirm that the mini release, full-text datasets, abstracts, TLDRs, authors, papers, and binary indices are all available before testing with local models such as Ollama.
  • Maintenance — Rebuild indices after changing local dataset files, verify files after an interrupted download, or inspect which release Valsci will use.
  • Readiness checks — Confirm that a Semantic Scholar API key, local release, and binary indices are available before running claims against real local data.

Semantic Scholar Readiness

Valsci always uses the real Semantic Scholar retrieval path. The Data page shows whether the pieces needed for that path are present.

  • Semantic Scholar API key — Live search and dataset downloads need SEMANTIC_SCHOLAR_API_KEY in app/config/env_vars.json.
  • Local release — Paper content lookup uses the latest local release unless a run selects another release.
  • Binary indices — Dataset rows are found through corpus-ID and author-ID indices. Missing indices should be rebuilt before a rehearsal run.

The app can start without the API key so the Data page remains available for setup and inspection, but live search and downloads will fail until the key is configured.

Local Data State Panel

The top-left panel summarizes the machine's Semantic Scholar setup:

  • Latest Release — The newest locally detected release ID, such as 2026-05-26-mini-mendelian-v1.
  • Local Releases — Count of release folders and indexed releases detected under the local Semantic Scholar dataset directory.
  • Selected Size — Disk usage for the release currently selected in the Releases panel.
  • S2 API Key — Shows whether SEMANTIC_SCHOLAR_API_KEY is configured. Downloads and live search require it.
  • Status card — Warns when no release is present, the API key is missing, or setup work is still needed.
  • Fetch missing papers & author data from the web — A toggle. When on, if a paper or author a claim needs isn't in your local corpus, Valsci pulls it from the live Semantic Scholar API (paper abstract/TLDR, and author h-index/citation counts for bibliometrics), using your API key and respecting rate limits. This is most useful with a small mini corpus, where live search routinely returns papers outside the bundled set. It takes effect for processing within a few seconds — no restart needed.
  • Refresh — Re-scans local files, indices, manifest state, and data jobs.

Releases Panel

The Releases panel lets you inspect one local Semantic Scholar release at a time.

  • Release selector — Choose among locally detected releases. Mini releases are labeled as mini.
  • New Release — Opens a wizard for creating either the bundled Mendelian mini release or a selected full Semantic Scholar release.
  • Type — Shows whether the selected release is a curated mini release or a full release.
  • Size — Total size of dataset JSON files under that release.
  • Topic — For manifest-backed mini releases, shows the topic label such as mendelian_disease_v1.
  • Index — Summarizes whether the selected release has the binary indices needed for local lookup.
  • Records written pills — Mini releases show counts for compact extracted datasets such as papers, authors, abstracts, TLDRs, and S2ORC v2 records.
  • Manifest coverage card — Only appears for the bundled mini release. It compares local compact rows against the current tracked manifest. Full Semantic Scholar releases do not use a manifest; their readiness is based on downloaded dataset files and indices.
  • Release Maintenance — Lets you verify the selected release or index/reindex selected datasets.

Dataset Coverage Panel

This table is the fastest way to see whether the selected release can support production-style evidence lookup.

  • Dataset — The Semantic Scholar dataset name and label. Valsci expects papers, authors, abstracts, TLDRs, and S2ORC v2.
  • Local Files — Shows whether JSON files exist for that dataset and how many local shard files were found.
  • Size — Total disk footprint for that dataset in the selected release.
  • Index — Shows binary index state by ID type, such as corpus ID or author ID. Missing index badges mean lookup will be slower or unavailable for that dataset.
  • Role — Describes how the dataset is used. S2ORC v2 provides local full text, abstracts and TLDRs provide fallback text, papers provide metadata, and authors provide bibliometrics.

New Release Wizard

The New Release wizard starts downloader and indexing jobs. Output appears in Current Job, and completed jobs appear in Recent Data Jobs.

  • Curated Mini — Runs the manifest-backed Mendelian mini builder. It reads the bundled ID manifest at semantic_scholar/manifests/mendelian_v1.json, streams Semantic Scholar dataset shards for the fixed corpus and author IDs, writes compact dataset-shaped JSONL files into the ignored runtime datasets directory, and builds binary indices.
  • Download Full Dataset Selection — Downloads selected full Semantic Scholar datasets for the requested release. This can consume a large amount of disk space, especially when S2ORC v2 is selected.

Mini Manifest

When Curated Mini is selected, the Mini Manifest input points to the tracked Mendelian ID manifest, normally semantic_scholar/manifests/mendelian_v1.json. Runtime release output and any source-row cache still go under semantic_scholar/datasets, which is ignored local data.

Full releases do not have or need a manifest. For a full release, Valsci downloads the selected Semantic Scholar datasets for the requested release ID and verifies them against the Semantic Scholar Datasets API file list.

Release

For full downloads, the Release input selects the Semantic Scholar release to download. Leaving it as latest uses the downloader's current latest-release behavior. Mini builds take their mini release ID from the manifest.

Datasets

The dataset picker controls which Semantic Scholar datasets a full release download touches. Selected-release verification and indexing use the dataset picker inside Release Maintenance.

Current Job Panel

  • Status badge — Shows idle, queued, running, success, failed, cancelled, or interrupted.
  • Command — Displays the exact Python downloader command the app launched.
  • Log stream — Shows downloader output while the job runs. It scrolls as new lines arrive.
  • Cancel Job — Requests cancellation for queued or running jobs.

Recent Data Jobs Panel

Recent jobs show their operation, status, creation time, and command. Use this history to confirm what was run after a long download or to inspect a failed command.

Recommended First Real-Data Workflow

  • Open Data and confirm S2 API Key is configured.
  • Click New Release, choose Curated Mini, and confirm the Mendelian mini manifest path.
  • Start the job and watch Current Job until it succeeds.
  • Select the new mini release in Releases.
  • Check Dataset Coverage for papers, authors, abstracts, TLDRs, S2ORC v2, and indices.
  • Run a small claim batch using the same local Ollama model you plan to use in production rehearsal.

Troubleshooting

  • Semantic Scholar API key is missing — Set SEMANTIC_SCHOLAR_API_KEY in app/config/env_vars.json before live search or downloads.
  • Mini release needs rebuild — The tracked manifest changed after the selected mini release was built. Use New Release → Curated Mini, or run python -m semantic_scholar.utils.downloader --mini.
  • Mini manifest missing — Restore the tracked mini package at semantic_scholar/manifests/mendelian_v1.json, or pass a custom manifest path.
  • Indices missing — Use Index or Reindex in Release Maintenance for the affected release and datasets.
  • Full download feels too large — Use the curated mini corpus for desktop rehearsal. Full S2ORC datasets are large and can exceed practical local disk limits.
  • Live search misses curated papers — Adjust the curated claims or manifest selection, then rebuild the mini corpus. The mini data is deterministic, but live search ranking can still change.

Claim Detail

View detailed results for a single claim and manage reruns.

You reach this page by clicking a claim from the Claims Browser, Batch Results, or Arena Workspace.

Focused Run

The hero section at the top displays the currently focused run's key information: claim text, rating, status, provider, model, cost, and timestamps. If a specific run was selected (e.g., from an arena), that run is shown. Otherwise, the most recent run is displayed.

  • Open Report / Open Arena Stage — Navigate to the full Claim Report page for this run.
  • Open Trace — Open the Trace View to see every step of this run's processing.

Runtime and Health

This panel summarizes execution state and warning signals: current stage, elapsed time, token usage, and counters for issues, retries, truncation, context warnings, and inaccessible papers. When the issue count is non-zero it becomes a link to the Issues & Warnings panel below.

Issues & Warnings

If anything was logged while the run processed, this panel lists each record — severity, stage, message, timestamp, and details — with a Download (.jsonl) button for the raw log. It opens with a banner that interprets the records against the run's final outcome, so you can tell recovery from real failure at a glance:

  • Severity tells you what happened next. A WARN is a transient failure that was automatically retried (e.g. "failed on attempt 1 of 3 — retrying"). An ERROR is terminal — the step gave up after exhausting retries.
  • The banner ties it to the verdict. "This run completed despite the errors below" (the run still produced a verdict) versus "This run did not complete" (no verdict — the errors explain why). A run can log errors and still succeed, because most failures are retried.

Stage Output

If the focused run has intermediate stage data (e.g., from an arena that stopped after a middle stage), it is displayed in this collapsible section.

Alternative Runs

A list of all other runs for this claim. Each card shows the provider, model, status, rating, and cost. Actions:

  • Focus Run — Switches the hero section to display this run's details.
  • Compare checkbox — Select multiple runs to include them in the Comparison Matrix below.
  • Detail — Opens the run in a new context.

Rerun This Claim

A form to queue a new evaluation of the same claim (or a modified version). The controls mirror the Home Page's run configuration: provider, model, duplicate strategy, batch tags, and advanced settings (queries, results per query, bibliometrics).

  • Review & Queue Rerun — Estimates the cost of re-evaluating this claim, then asks you to approve the spend in a confirmation modal before queueing.

Comparison Matrix

A side-by-side table comparing all runs selected via the compare checkboxes. Shows each run's rating, evidence count, cost, and other key metrics in columns for easy visual comparison.

Tip: The Comparison Matrix is one of Valsci's most powerful features. If two models give the same claim different ratings, compare their evidence side-by-side to understand why. Did one find papers the other missed? Did they interpret the same excerpts differently? This helps you calibrate which models you trust for specific domains.

Claim Report

The full evaluation report for a single claim run.

You reach this page by clicking "View Report" from Batch Results or "Open Report" from Claim Detail.

Report Content

The report displays:

  • Claim text and overall rating with a color-coded badge (red for contradicted, green for highly supported, gray for no evidence or mixed).
  • Explanation — A markdown-formatted narrative summarizing the evidence and reasoning behind the rating.
  • Final Reasoning — If available, a collapsible section with the LLM's detailed reasoning. Click Show Final Reasoning / Hide Final Reasoning to toggle.

Papers Sections

  • Relevant Papers — Papers that the LLM found relevant to the claim. Each entry shows:
    • Title — The paper's title as retrieved from Semantic Scholar.
    • Authors — Author names with their H-index (a measure of research impact; higher is more influential).
    • Relevance — The LLM's explanation of why this paper is relevant to the claim.
    • Bibliometric Impact — A composite reliability score (when bibliometrics are enabled) combining author, citation, and venue quality.
    • Excerpts — Key passages from the paper that support or contradict the claim.
    • Explanations — The LLM's interpretation of how each excerpt relates to the claim.
    • Citations — Formatted citation entries for the paper.
    • Content Type — The type of paper (e.g., research article, review, meta-analysis).
    • Read Paper — A link to the original paper on Semantic Scholar.
  • Other Reviewed Papers — Papers that were retrieved but determined not relevant, with an explanation of why.
  • Inaccessible Papers — Papers that could not be retrieved or analyzed, with the reason.

Additional Information

  • Search Queries — The actual search queries generated by the LLM and sent to Semantic Scholar.
  • Processing Time Breakdown — How long each phase took (search, paper processing, report generation, total).
  • LLM Usage — Token counts (input, output, total) and estimated cost.
  • Usage By Stage — A table breaking down token usage and cost per pipeline stage.
  • Prompt Provenance — Expandable section showing which prompt files were used, their SHA-256 hashes, and modification dates. Useful for reproducibility.
  • Debug & Trace — Summary of LLM calls, models used, retries, and a link to the full Trace Viewer.

Action Buttons

  • Download Report — Download the report as a Markdown file.
  • View Full Trace — Open the Trace View page for this run.
  • Download Trace — Download the raw trace file (JSONL, gzipped).
  • Download Issues — Download the issues log (JSONL, gzipped).
  • Download Citations — Download citation data as a JSON file.
  • Delete Claim — Permanently delete this claim and all its data. Requires confirmation.
Tip: The Prompt Provenance section is valuable for reproducibility. If you modify prompt templates and re-run a claim, the SHA-256 hashes let you verify exactly which prompt version produced which result. This is especially useful in team settings or when iterating on prompt engineering.

Batch Results

Shows the batch status (processing or completed), the number of completed claims, and counts of saved vs. queued claims.

Action Bar

  • Delete Batch — Permanently delete the entire batch and all its claims. Requires confirmation.
  • Download CSV — Export the results table as a CSV file.
  • Bulk Download Reports — Download all claim reports as a ZIP of Markdown files. Only available after the batch is complete.

Results Table

  • Claim — The claim text.
  • Status — Processing status (completed, processing, error).
  • Location — Where the claim data is stored (saved_jobs or queued_jobs).
  • Rating — The numerical claim rating (0-5).
  • Supporting / Non-Relevant — Count of relevant and non-relevant papers found.
  • Price — Estimated cost for that claim's evaluation.
  • Report — Links to either View Report (if completed) or View Progress (if still processing), plus a Claim link to the Claim Detail page.

Total Batch Price

The sum of all claim costs in the batch, displayed below the table.

Tip: Use column header filters to quickly find claims by status (e.g., type "error" in the Status filter to find failed evaluations). Click any column header to sort—sorting by Rating helps you quickly find the most and least supported claims.

Progress

Real-time processing status for a batch or individual claim.

You are sent here automatically after submitting claims. The page refreshes every five seconds (a countdown in the header shows when the next refresh lands) and displays:

  • Animated progress bar — Fills as claims complete; an animated stripe pattern indicates the batch is actively being worked.
  • Pipeline stages — Counts of claims at each step (Queued → Searching literature → Analyzing evidence → Done), with a separate chip for errors.
  • Current claim — Which claim is currently being evaluated, plus elapsed time on the page.
  • Processor health — If the background processor daemon is not running, a banner warns you that claims will stay queued and explains how to start it. The processor reports a heartbeat every few seconds while alive.

Once all claims are complete, the page automatically redirects to the results (the claim report for a single claim, or Batch Results for a batch). If the batch settles with failed claims, a warning summarizes how many failed so you can inspect or retry them.

Tip: Processing time depends on the number of claims, the number of queries per claim, and the speed of your chosen LLM provider. A single claim with default settings (5 queries, 5 results each) typically takes 30 seconds to a few minutes. Local models (Ollama/llama.cpp) may be slower depending on your hardware.

Arena Builder

Create a multi-model comparison arena.

Navigate here using New Arena from the Home page or the Arenas nav link followed by clicking "New Arena."

When to Use an Arena

Arenas are best when you need to:

  • Compare models — See how GPT-4o, Claude, Llama, etc. evaluate the same evidence differently.
  • Benchmark cost vs. quality — A cheaper model might produce results nearly as good as a premium one for your domain.
  • Build confidence — If three independent models all rate a claim "Highly Supported," that's stronger evidence than one model's opinion.
  • Control variables — Use "Reuse retrieval" mode to give all models the same papers, isolating the analysis quality from retrieval differences.

A four-step visual guide at the top of the page: Step 1 Choose provider, Step 2 Choose models, Step 3 Add claims, Step 4 Review and launch.

  • Model search — Filter the model list by typing.
  • Model checkboxes — Check each model you want to include. Each checked model becomes a candidate.
  • Remove — Remove this provider block (only shown when multiple blocks exist).

Claims

  • Upload Claims — Load from a .txt file.

Understanding the Arena Lifecycle

An arena progresses through this cycle:

  • Launch — All candidates process all claims through the first stage (or up to the "Stop After" setting).
  • Review — The arena pauses. You examine results in the Overview and Claims tabs, select winners, and skip claims you don't want to continue.
  • Continue — Selected claims and candidates advance to the next stage. The arena processes them and pauses again for review.
  • Complete — When all stages are done (or you choose not to continue further), the arena is marked as completed. All results remain accessible for future reference.

Providers

Configure LLM provider connections and manage models.

Navigate here using the Providers link in the top navigation bar.

Providers vs. Settings (which wins). This page is the source of truth for which providers and models runs actually use. The provider env values on the Settings page (LLM_PROVIDER, LLM_BASE_URL, LLM_API_KEY, LLM_EVALUATION_MODEL) only seed the "Default Config Provider" here. A model's context window and max output tokens set here take precedence over the LOCAL_MODEL_CONTEXT_OVERRIDE and MODEL_REGISTRY_OVERRIDES fallbacks in Settings (and for Ollama, the live model's window auto-detected from /api/show wins over both). Per-task output budgets are separate and live in Settings → Model Routing & Token Budgets.

Understanding Provider Types

Valsci supports several LLM provider types, each with different trade-offs:

  • OpenAI — The most common choice. Connects to OpenAI's API (GPT-4o, GPT-4, etc.). Requires an OpenAI API key. Models are high quality but incur per-token costs.
  • OpenRouter — A unified API gateway that provides access to models from many providers (OpenAI, Anthropic, Meta, Google, etc.) through a single API key. Useful if you want to compare models across providers without managing multiple API keys.
  • Ollama — Run open-source models (Llama, Mistral, Gemma, etc.) locally on your own hardware. Free to run but requires sufficient GPU/CPU resources. Valsci can auto-discover which models are installed on your Ollama instance.
  • llama.cpp — Connect to a local llama.cpp server. Similar to Ollama but uses the llama.cpp HTTP server directly.
  • Azure OpenAI — OpenAI models hosted on Microsoft Azure. Requires an Azure OpenAI resource endpoint and API key. Used by organizations with Azure enterprise agreements.
  • Azure AI Inference — Azure's model inference endpoint for non-OpenAI models hosted on Azure. Requires a separate endpoint URL.

Provider Sidebar

The left panel lists all available provider types. Each row has:

  • Toggle switch — Enable or disable this provider type. Disabled providers don't appear in the Provider dropdowns on other pages.
  • Click the row — Select this provider type for editing in the right panel.

Provider Editor

The right panel shows the configuration for the selected provider. An Unsaved changes badge appears when you have modified settings that haven't been saved yet.

Connection Settings

  • Label — A display name for this provider (shown in dropdowns across the app).
  • API Key — The authentication key for the provider's API. Stored locally in the provider catalog file.
  • Base URL — Only needed if the provider uses a non-default API endpoint (e.g., a proxy or self-hosted deployment).

Additional fields appear depending on the provider type:

  • Local Backend (Local provider) — Select between Ollama and llama.cpp backends.
  • HTTP Referer / Site Name (OpenRouter) — Optional headers sent with OpenRouter API requests.
  • Azure OpenAI Endpoint / API Version (Azure OpenAI) — Your Azure resource endpoint and API version.
  • Azure AI Inference Endpoint (Azure AI Inference) — Your Azure AI inference endpoint URL.

  • Discover From Provider URL — Queries the provider's configured base URL to find all available Ollama models.
  • Probe Alternate URL — Queries a different URL (entered in the Alternate URL field) to find models. Useful if Ollama is running on a different port or host.

Discovered models appear in an import modal where you can filter, select, and add them to the provider.

  • Select Visible / Clear Visible — Bulk check or uncheck all models matching the current filter.

  • Model Name — The identifier sent to the API (e.g., gpt-4o, llama3:latest).
  • Label — Optional human-friendly name shown in the UI.
  • Context Window — Maximum input token capacity.
  • Default Max Output — Default limit on output tokens per request.

Settings

Edit Valsci's configuration (env_vars.json) from the browser.

The Settings page edits the values in app/config/env_vars.json. Settings are organized into collapsible groups (AI Provider, Model Routing & Token Budgets, Performance & Rate Limits, Literature Data, Diagnostics & Tracing, Email, Security, General, and Storage Paths). Each setting shows a friendly label, its raw key, a description, and badges for whether it is Configured (set in your file) or running on a Default.

Finding and editing settings

  • Search — Filter all settings by name or description.
  • Only configured — Show just the settings you have explicitly set.
  • Save Changes — Writes every change back to env_vars.json. A Unsaved changes badge appears while you have pending edits.
Restart badge. Saving applies changes to the running web app immediately, but the background processor reads its configuration at startup. Settings marked Restart only affect claim processing after you restart the processor (e.g. docker compose restart processor).

Per-task output token budgets

Under Model Routing & Token Budgets, the Task Routing & Budgets setting includes a friendly editor with one field per pipeline stage (Query Generation, Paper Analysis, Venue Scoring, Final Report). Each field sets the maximum number of tokens a model may generate for that stage.

This matters for reasoning ("thinking") models: they spend output tokens on hidden chain-of-thought before answering, so a budget that is too small gets used up before any answer is produced, and the run fails with empty output. If you see that error, raise the budget for the named stage (e.g. 16000) here. Leave a field blank to use the built-in default. The full routing object is still editable as raw JSON under Advanced.

Structured-output reliability

Also under Model Routing & Token Budgets, two settings control how strictly Valsci holds models to the JSON shape it expects for the final report and query generation:

  • Strict JSON Schema (on by default) — Forces those stages to match an exact JSON schema instead of just "valid JSON," so a model can't invent its own field names. If a provider doesn't support schema mode, Valsci automatically falls back to plain JSON. Recommended on; especially helpful for local models that otherwise drift from the required fields.
  • JSON Repair Pass (off by default) — When a model returns valid JSON with the wrong fields, run a second LLM pass that reshapes its output into the required schema (without changing the content), then re-validate. It costs an extra call only when the first output fails validation. Turn it on if a particular model frequently produces non-conforming output even with strict schema.

Trace View

Inspect the detailed execution trace for a claim run.

You reach this page from the Claim Report ("View Full Trace") or Claim Detail ("Open Trace").

What Is a Trace?

A trace is a chronological log of every significant action during a claim's evaluation: each LLM call (with the exact prompts and responses), Semantic Scholar API requests, paper parsing steps, scoring calculations, and error recovery. It provides full transparency into how Valsci arrived at its verdict.

Trace Records

Each record in the trace is displayed as a row with:

  • Timestamp — When the action occurred.
  • Stage — Which pipeline stage (query generation, paper analysis, venue scoring, or final report).
  • Type — The kind of action (LLM call, API request, processing step, error, etc.).
  • Details — Expandable content showing full request/response data, token counts, model used, and timing.

Navigation

  • Highlighted records — Error-like records (failures, retries, warnings) are visually highlighted so you can quickly spot issues.
  • Focused record — If you arrived via a link from an issue banner (e.g., "View trace section"), the relevant record is scrolled into view and highlighted.
Tip: Traces are most useful for debugging unexpected ratings, understanding why certain papers were included or excluded, and verifying that prompts are working as intended.
